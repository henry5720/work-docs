# **2\. 系統概述**

## **專案背景**

為配合企業內部對表單資料簽核的需求，於 TeamSync 平台新增「資料簽核模組」。本模組與核心功能「AI 聊天室」分工協作：使用者於聊天室以自然語言將資料寫入表格，若該表格已啟用簽核，資料不會立即成為可用資料，而是先進入審核流程；待審核通過後，才由程式寫回表格成為正式（可搜尋、可匯出、可統計）資料。

簽核模組本身提供三個入口：「審核流程設定」（獨立模組，建立可重用的審核流程模板）、「審核中心」（右上角固定入口，含待我簽核與我送出的兩頁）、以及系統管理下的「群組標籤」（將人員分組，供簽核與其他模組權限套用）。表格端則於「表格建立精靈」與「表格設定 · 審核」選擇要套用的流程。

## **使用者情境（User Story）**

* 作為簽核流程管理者（中階以上權限），我希望自訂多關卡的審核流程，於每一關指定審核者（個人、群組標籤或動態角色）與通過條件（單人／任一人／所有人），並將此模板套用到不同表格或 RPA 節點。

* 作為審核者，我希望在審核中心一覽所有待我簽核的項目，逐筆查看詳情後核准、否決或退回，也能對多筆一次批次通過或批次否決。

* 作為申請者，我希望在「我送出的」查看每一筆送審的狀態（審核中／通過／退回／否決）與結果通知，若被退回或否決則重送一筆新資料。

* 作為系統管理員，我希望以群組標籤將人員分組（可跨部門、一人可屬多組），供簽核流程與其他模組權限套用。

## **設計原則**

* **判斷與執行分離：**審核者的決定只更新審核狀態；資料寫回正式表格由確定性程式於「審核通過」時執行。

* **流程模板化：**審核流程為可重用模板，一份流程可套用於多張表格；表格端只綁定流程，不各自定義關卡。

* **送審快照（snapshot）：**每筆送審單於建立當下凍結流程副本並解析動態角色為實際帳號；日後模板變更、主管異動皆不影響進行中的單，並保留稽核軌跡。

* **資料狀態閘門：**僅狀態為「無須審核」或「審核通過」的資料可被 AI 聊天室搜尋、匯出與統計；「審核中／已退回／已否決」僅顯示於表格畫面。

* **終結即重送：**否決與退回申請人皆為終結態，該筆資料不可原地編輯，使用者須重寫一筆新資料送審；退回上一關則非終結，資料仍在流程中。

* **併發一致性：**同一關卡多位審核者時，以「第一個送出決定者」為準（first submitted decision wins），其餘人之待簽項目結算後即移除或轉唯讀，後送決定一律擋下。

* **防呆優先：**執行中／已套用的流程不可刪除或變更；有「審核中」資料的表格不可切換或關閉流程；影響進行中審核的群組成員異動須先檢查。

## **範圍邊界（Out of Scope）**

* 本期僅支援線性（垂直）審核流程；不含平行審核（分支同時進行）。資料結構為關卡陣列，暫不預留平行分支。

* 不含申請者「主動撤回」尚未完成的送審。

* 不含退回後原地編輯：被退回或否決之資料一律重送新單，不提供修改後重送同一筆。

* 動態角色本期為：申請人的直屬主管、職務代理人、次要職務代理人、部門主管；其解析規則見 C002。

* 不含簽核與外部系統（HR、財務）之整合；以事件／API 對接，不內建。

# **3\. 新增介面與角色定義**

簽核模組的介面由四處構成：獨立的「審核流程設定」模組、右上角固定的「審核中心」、系統管理下的「群組標籤」，以及聊天室表格端的「審核設定」（建表精靈與表格設定）。AI 聊天室為資料寫入與送審的發起入口。

| 頁面 / 分區 | 內容 | 使用者 / 所需權限 | 讀寫 |
| :---- | :---- | :---- | :---- |
| 審核流程設定（獨立模組） | 建立 / 編輯 / 刪除審核流程模板，設定關卡、審核者、通過條件、fallback，並套用到表格。 | 審核流程管理者（中階以上管理權限） | 可寫 |
| 審核中心 · 待我簽核 | 列出所有待該使用者處理之簽核，逐筆核准 / 否決 / 退回，或批次通過 / 批次否決。 | 審核者 | 可寫（決定） |
| 審核中心 · 我送出的 | 查看自己送出之簽核狀態與結果通知（審核中 / 通過 / 退回 / 否決）。 | 申請者 | 唯讀 |
| 系統管理 · 群組標籤 | 建立群組標籤、指派成員；可跨部門，一人可屬多組。 | 中階以上管理者 | 可寫 |
| 表格設定 · 審核 | 為既有表格選擇 / 切換 / 關閉套用的審核流程。 | 具表格設定權者 | 可寫 |
| 表格建立精靈 · 權限&審核 | 建表時於此步驟選擇要套用的審核流程（可略過）。 | 建表者 | 可寫 |

*（AI 聊天室：資料寫入與送審發起入口。使用者以自然語言寫入表格，系統依表格是否啟用審核決定資料直接落帳或先送審；不在本模組頁面導覽內。）*

## **角色定義**

* **審核流程管理者：**具中階以上管理權限，於審核流程設定建立、編輯、刪除流程模板，並將流程套用到表格。

* **審核者：**被指定於某關卡的個人、群組成員或動態角色所解析出的帳號；於審核中心對待我簽核之項目做決定。

* **申請者：**於聊天室寫入須審核資料而觸發送審者；於「我送出的」查看狀態與結果。

* **群組標籤：**人員分組單位，可作為審核者，亦可套用於其他模組權限之篩選；可跨部門，單一帳號不限單一群組。

* **動態角色：**不指定固定帳號，而於送審當下依申請人之帳號關係解析（直屬主管、職務代理人、次要職務代理人、部門主管）。

# **5\. 使用者操作流程**

## **5.1 總流程**

初始設定（一次性） ➔ 日常使用（聊天室送審 / 審核中心處理 / 表格檢視）

* 初始設定：建立群組標籤並指派成員（C001） ➔ 於審核流程設定建立流程模板、設定各關卡審核者與條件（A001） ➔ 於建表精靈或表格設定將流程套用到表格（D001）。

* 日常使用：申請者於聊天室寫入須審核資料 ➔ 系統建立送審單並進入審核流程 ➔ 審核者於審核中心逐關處理（B001 / B002） ➔ 通過後資料寫回正式表格、被退回 / 否決則通知申請人重送（D003）。

## **5.2 送審共用骨幹（D001 → D002）**

* 使用者於聊天室以自然語言寫入某表格之資料。

* 系統判斷該表格是否啟用審核並已比對到流程：未啟用 ➔ 資料狀態＝「無須審核」，直接寫入正式表格；已啟用 ➔ 建立 ApprovalRequest，資料狀態＝「審核中」，並於建立當下凍結流程快照、解析動態角色為實際帳號（C002）。

* 進入第一關，依關卡審核者與通過條件等待處理；「審核中」之資料僅顯示於表格，不進入 AI 搜尋 / 匯出 / 統計（D002 閘門）。

## **5.3 審核者決定流程（B001，含關卡結算）**

* 審核者於「待我簽核」點選項目，開啟詳情：申請人、資料來源、申請時間、詳細資料、展開此審核流程、審核備註。

* **單人核可：**該審核者核准 ➔ 進入下一關；否決 / 退回申請人 ➔ 終結；退回上一關 ➔ 回退。

* **任一人核可（OR）：**以第一個送出決定者為準——核准 ➔ 整關通過進下一關（E1）；否決 / 退回申請人 ➔ 整筆終結；退回上一關 ➔ 整筆回退。結算後其餘人之待簽項目移除，後送決定擋下並提示「已由 XXX 處理」。

* **所有人核可（AND）：**須全員核准才進下一關；任一人否決 / 退回申請人 ➔ 整筆終結；任一人退回上一關 ➔ 整筆回退。

* 過最後一關 ➔ 狀態改為「審核通過」，資料寫回正式表格並通知申請人（D003）。

## **5.4 退回與否決處理（B001 / D003）**

三種負面決定的去向不同，須分開處理：

| 動作 | 資料去向 | 是否終結 | 後續 |
| :---- | :---- | :---- | :---- |
| 否決 | 終結整筆，狀態＝已否決 | 是（terminal） | 帶原因通知申請人；申請人重送新單。 |
| 退回申請人 | 終結整筆，狀態＝已退回 | 是（terminal） | 帶原因通知申請人；申請人重送新單（不可原地編輯）。 |
| 退回上一關 | 回到上一關重審，狀態維持審核中 | 否 | 通知上一關審核者重新處理；上一關再核可後接回原本這關續跑。 |

* 否決與退回申請人行為一致（皆終結 \+ 重送），前端保留兩顆按鈕僅為給申請人不同訊號（否決＝不准；退回＝方向可以、補正後再送），後端狀態機同走終結。

* 否決與退回時，原因為必填（單筆於備註強制填寫；批次否決要求共同原因）。

* 第一關無上一關，隱藏「退回上一關」，僅保留「退回申請人」（B3）。

## **5.5 資料狀態與檢視流程（D002）**

* 資料狀態＝「無須審核」或「審核通過」：可於 AI 聊天室搜尋、查找、匯出、統計分析。

* 資料狀態＝「審核中」「已退回」「已否決」：不可被搜尋 / 匯出 / 統計，僅顯示於表格畫面之欄位中，並可用資料狀態 filter 篩選。

# **6\. 功能詳述與驗收標準**

### **A001 \- 建立指定審核流程**

**目標結果：**於審核流程設定建立可重用的審核流程模板，套用於不同表格或 RPA 節點。

**功能規則：**

* 中階以上管理者可自由新增 / 刪除流程模板，無數量限制。

* 每個流程由一個以上關卡組成（線性排列，可拖曳調整順序）；每關設定：關卡名稱、審核者、通過條件、找不到對應人時之處理。

* 審核者可為個人帳號、群組標籤或動態角色（直屬主管 / 職務代理人 / 次要職務代理人 / 部門主管）。

* **通過條件依審核者組成決定顯示：**單一個人 ➔ 固定為「單人核可」且隱藏切換；含群組或審核者複數 ➔ 顯示切換，可選「任一人核可（OR）」或「所有人核可（AND）」，預設任一人核可。

* 找不到對應人（fallback，僅動態角色關卡有意義）：不可送出（提醒補齊對應人）／改送備援審核者／跳過此關。實際觸發依 supervisorStatus，詳見 C002。

**驗收標準（AC \- 條件切換顯示）**

**Given** 某關卡審核者僅為單一個人「王小明」。

**When** 管理者於該關卡設定通過條件。

**Then** 系統固定為「單人核可」且不顯示 OR / AND 切換；當再加入第二位審核者或群組時，才出現切換並預設「任一人核可」。

**驗收標準（AC \- 線性流程）**

**Given** 管理者建立一個三關流程。

**When** 拖曳調整關卡順序並儲存。

**Then** 關卡以線性順序重排並套用；系統不提供平行分支設定。

### **A002 \- 流程刪除 / 變更防呆**

**目標結果：**避免刪除或變更正在被使用的流程而破壞進行中之審核。

**功能規則：**

* 流程若已套用至任何表格，或有任何引用此流程之「審核中」送審單，則不可刪除、不可變更關卡結構。

* 即使流程被變更或停用，進行中的送審單仍依其送審快照走完，不受影響（見附錄 A 之 ApprovalRequest）。

**驗收標準（AC \- 執行中禁刪）**

**Given** 「報帳簽核流程」已套用於表格且有 2 筆審核中資料。

**When** 管理者嘗試刪除或修改該流程關卡。

**Then** 系統阻擋並提示流程使用中不可刪除 / 變更；既有審核中資料續依快照進行。

### **B001 \- 審核中心 · 待我簽核（做決定）**

**目標結果：**審核者於審核中心確認需審核之資料並做出決定。

**功能規則：**

* 審核中心位於右上角使用者頭像側；「待我簽核」列出所有須協助審核之項目，可依申請來源、緊急度、申請人篩選。

* 點選項目開啟詳情：申請人、資料來源、申請時間、詳細資料、展開此審核流程、審核備註。

* 可執行：核准通過、否決、退回上一關、退回申請人。否決 / 退回須填原因。

* 第一關隱藏「退回上一關」。關卡結算依通過條件（單人 / OR / AND，見 5.3）。

* APP 端亦須提供審核中心，供審核者 / 申請者以手機操作。

**驗收標準（AC \- 否決需原因）**

**Given** 審核者欲否決某筆申請。

**When** 未填寫原因即按否決。

**Then** 系統擋下並要求先填原因；填妥後方可送出，狀態改為已否決並通知申請人。

**驗收標準（AC \- 第一關退回）**

**Given** 某筆資料目前在第 1 關。

**When** 審核者開啟退回選單。

**Then** 僅顯示「退回申請人」，不顯示「退回上一關」。

**驗收標準（AC \- OR 先送者勝）**

**Given** 第 2 關為群組「財務審核」且條件為任一人核可，A、B 皆為審核者。

**When** A 先送出核准，B 稍後才開啟同一筆欲否決。

**Then** 該關以 A 核准結算並進入下一關；B 之待簽項目已移除，其後送決定被擋下並提示已由 A 處理。

### **B002 \- 審核中心 · 批次操作**

**目標結果：**對多筆待簽核項目一次處理。

**功能規則：**

* **批次通過僅代表「通過我這一關」：**被選項目各自依所在關卡前進至下一關或寫入正式表格，不得將多關卡項目直接標為整案通過。

* 批次否決須填寫共同原因方可送出（與單筆否決一致）；否決後各筆流程直接結束並帶原因通知申請人。

**驗收標準（AC \- 批次通過語意）**

**Given** 選取兩筆待簽核，均為三關流程之第 1 關。

**When** 執行批次通過。

**Then** 兩筆狀態皆前進至第 2 關（非「已通過」），仍為審核中。

**驗收標準（AC \- 批次否決需原因）**

**Given** 選取三筆待簽核並按批次否決。

**When** 未填共同原因。

**Then** 系統擋下並要求填原因；填妥後三筆皆否決並各自通知申請人。

### **B003 \- 審核中心 · 我送出的**

**目標結果：**申請者查看自己送出之簽核狀態與結果通知。

**功能規則：**

* 列出申請者所有送審紀錄，顯示狀態（審核中 / 審核通過 / 已退回 / 已否決）與當前關卡，可 filter。

* 被退回 / 否決者可查看原因；本期不提供主動撤回，被退回 / 否決須重送新單。

**驗收標準（AC \- 狀態與原因）**

**Given** 申請者有一筆被退回之送審單。

**When** 於「我送出的」查看該筆。

**Then** 顯示狀態為已退回並可查看退回原因；無撤回按鈕。

### **C001 \- 群組標籤功能**

**目標結果：**管理員可建立群組標籤並指派成員，供簽核與其他模組權限套用。

**功能規則：**

* 中階以上管理者於系統管理 · 群組標籤新增 / 編輯群組並指派成員；群組可跨部門，單一帳號不限單一群組。

* 已套用（被流程 / 表格引用）之群組不可刪除；刪除鈕於此情況停用。

* **成員異動防呆（D2）：**移除群組成員或刪除群組前，須檢查該成員 / 群組是否正擔任某「審核中」送審單之當前待審者；若是則阻擋或提示，避免該筆審核無人可審而卡住。

**驗收標準（AC \- 已套用禁刪）**

**Given** 群組「財務審核」已被報帳簽核流程引用。

**When** 管理者嘗試刪除該群組。

**Then** 系統阻擋並提示已套用不可刪除。

**驗收標準（AC \- 成員異動影響審核）**

**Given** 某筆審核中資料之當前待審者為群組「財務審核」，且該群組僅餘一位可審成員。

**When** 管理者嘗試將該成員移出群組。

**Then** 系統偵測到將導致該筆無人可審並阻擋 / 提示，須先處理該筆或改派後方可移除。

### **C002 \- 動態審核者解析（supervisorStatus 三態 \+ fallback）**

**目標結果：**含動態角色之關卡，於送審當下解析為實際審核帳號並寫入快照。

**功能規則：**

* 帳號主檔具 supervisorStatus 三態：assigned（已指定主管）、none（確認不需要主管）、unset（尚未設定）。

* 送審時針對「申請人的直屬主管」等動態角色解析：

  * **assigned** ➔ 帶入該帳號之主管（可能複數，依關卡 OR / AND 條件處理）。

  * **none** ➔ 自動跳過此關，流程接續下一關（不論關卡 fallback 設定）。

  * **unset** ➔ 觸發關卡 fallback，預設「不可送出，提醒具帳號管理權限者補齊」；管理者亦可設為「改送備援審核者」或「跳過此關」。

* 解析結果（實際帳號）寫入送審快照；日後主管異動不影響進行中之單。

**驗收標準（AC \- none 自動跳過）**

**Given** 申請人 supervisorStatus 為 none，第 1 關審核者為「申請人的直屬主管」。

**When** 送出審核。

**Then** 第 1 關自動跳過，直接進入下一關。

**驗收標準（AC \- unset 擋下提醒）**

**Given** 申請人 supervisorStatus 為 unset，關卡 fallback 為「不可送出，提醒補齊」。

**When** 送出審核。

**Then** 系統擋下送出並通知具帳號管理權限者補齊主管設定。

### **D001 \- 表格套用審核流程**

**目標結果：**於表格選擇要套用的審核流程；套用後新增資料先送審。

**功能規則：**

* 建表精靈「權限&審核」步驟與表格設定「審核」分頁皆可選擇流程；下拉可搜尋流程名稱或關卡。

* 若無可用流程，提示先至審核流程設定建立後再回來套用（建表時可略過）。

* 無論表格是否啟用審核，資料皆可透過自然語言正常寫入；差別在啟用審核者先進入送審。

**驗收標準（AC \- 套用生效）**

**Given** 表格「報帳紀錄表格」套用「報帳簽核流程」並儲存。

**When** 使用者於聊天室新增一筆報帳資料。

**Then** 該筆建立為審核中之送審單，未寫入正式可用資料。

### **D002 \- 資料狀態區隔與閘門**

**目標結果：**以「資料狀態」欄位區隔已審與未審資料，並據此控管可用性。

**功能規則：**

* 啟用審核之表格新增「資料狀態」欄位，值為：無須審核、審核中、已退回、已否決、審核通過，並提供 filter。

* 閘門：狀態 ∈ { 無須審核、審核通過 } ➔ 可被 AI 搜尋 / 匯出 / 統計；狀態 ∈ { 審核中、已退回、已否決 } ➔ 僅顯示於表格畫面。

**驗收標準（AC \- 閘門）**

**Given** 表格中有一筆審核中、一筆審核通過之資料。

**When** 於 AI 聊天室搜尋 / 統計。

**Then** 僅審核通過者納入結果；審核中者不被搜尋、匯出或統計，僅於表格顯示。

### **D003 \- 審核結果處理與通知**

**目標結果：**審核結果回到表格更新狀態，並通知資料寫入者。

**功能規則：**

* 通過（過最後一關）：資料狀態改為審核通過，寫回正式表格；於審核中心通知申請人（含審核者備註）。

* 退回申請人 / 否決：資料狀態改為已退回 / 已否決（終結），帶原因於審核中心通知申請人；申請人重送新單。

* 退回上一關：狀態維持審核中，通知上一關審核者重新處理。

**驗收標準（AC \- 通過寫回）**

**Given** 某筆資料通過最後一關。

**When** 最後一關核准。

**Then** 資料狀態改為審核通過並寫回表格，申請人收到通過通知。

**驗收標準（AC \- 否決帶因通知）**

**Given** 某筆資料於第 1 關被否決並填有原因。

**When** 否決送出。

**Then** 資料狀態改為已否決，申請人收到含原因之通知；此單結束，需重送新單。

### **D004 \- 表格切換 / 關閉流程防呆（in-flight）**

**目標結果：**避免在有進行中審核時切換或關閉表格所套用的流程，造成資料錯亂。

**功能規則：**

* **僅當表格存在「審核中」資料時鎖定：**不可切換所套用之流程、不可關閉審核。已通過 / 已退回 / 已否決等終結態不影響切換。

* 切換或關閉前，系統須先偵測是否有審核中資料；有則阻擋並提示先結清。

**驗收標準（AC \- 審核中鎖定）**

**Given** 表格有 3 筆審核中資料。

**When** 嘗試切換流程或關閉審核。

**Then** 系統阻擋並提示尚有審核中資料須先結清。

**驗收標準（AC \- 無審核中可切換）**

**Given** 表格僅有已通過與已退回資料，無審核中。

**When** 切換流程。

**Then** 允許切換，既有終結態資料不受影響。

# **7\. 異常處理與邊界條件（Edge Cases）**

1. **負面決定的三種去向：**否決與退回申請人皆為終結（terminal），該筆不可原地編輯、須重送新單；退回上一關非終結，資料維持審核中並回到上一關。

2. **OR 關卡併發：**同一關卡多審核者且條件為任一人核可時，以第一個送出決定者為準（先送者勝，非先看到者）；結算後其餘人待簽項目移除，後送決定擋下並提示已由他人處理。

3. **AND 關卡否決：**條件為所有人核可時，任一人否決 / 退回申請人即整筆終結（該條件已不可能達成）。

4. **第一關退回：**第一關無上一關，隱藏「退回上一關」，僅保留「退回申請人」。

5. **退回上一關遇跳過關：**若上一關曾因 supervisorStatus=none 或 fallback 而被跳過（無實際審核者），退回上一關須略過該空關，退到最近一個有實際審核者之前關；若前面已無有效關卡則等同退回申請人。

6. **退回上一關後續跑：**上一關重新核可後，接回原本退回的該關續審，非從頭重跑。

7. **零審核自動通過（可接受）：**若關卡因 supervisorStatus=none 或 fallback「跳過此關」被略過，且為唯一 / 最後一關，將導致無人審核即自動通過——本期視為可接受行為，惟建議管理者設定流程時避免。

8. **批次通過語意：**批次通過僅通過本關，各筆依所在關卡前進，不得將多關卡項目直接標為整案通過。

9. **批次否決需原因：**批次否決與單筆否決一致，須填共同原因方可送出。

10. **送審快照：**送審單於建立當下凍結流程副本並解析動態角色為實際帳號；模板日後變更 / 刪除、主管異動皆不影響進行中之單。

11. **流程刪除 / 變更防呆：**已套用或有審核中送審單引用之流程不可刪除或變更關卡結構。

12. **表格切換 / 關閉防呆：**表格存在審核中資料時，不可切換或關閉所套用之流程；切換 / 關閉前先偵測。

13. **群組刪除防呆：**已被流程 / 表格套用之群組不可刪除。

14. **群組成員異動防呆：**移除成員 / 刪除群組前，檢查是否為某審核中送審單之當前待審者，若將導致無人可審則阻擋或提示。

15. **資料狀態閘門：**僅「無須審核 / 審核通過」可被搜尋、匯出、統計；「審核中 / 已退回 / 已否決」僅顯示於表格。

16. **否決與退回申請人一致：**兩者後端皆走終結 \+ 重送，前端保留兩鈕僅為對申請人的訊號差異，須共用同一狀態轉移。

17. **通過條件顯示規則：**審核者為單一個人時固定「單人核可」且隱藏切換；含群組或複數審核者時才顯示 OR / AND，預設任一人核可。

18. **未涵蓋本期：**不支援平行審核；不支援申請者主動撤回；不支援退回後原地編輯。
