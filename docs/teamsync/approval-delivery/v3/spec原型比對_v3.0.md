# 資料簽核模組 v3.0 — 開發文件 vs UI 原型 比對報告

- 比對日期：2026/07/13
- 比對基準：`資料簽核模組_開發文件_v3.0.md`（2026/07/10）
- 原型：`teamsync-approval-delivery_v3/` 內 5 個 HTML 原型 + 1 份 UI 說明（ui-brief）
- 方式：逐頁細讀原型（含內嵌 JS 與示範資料），對照 spec 功能規則與驗收標準（AC）

## 結論摘要

原型與 spec 大方向一致（線性流程、送審快照、fallback 三選項、群組刪除防呆、supervisorStatus 三態等均有做且正確），但存在 **5 項行為直接違反 spec AC**、**4 項 spec 要求的防呆零實作**、**3 項資料契約對不上（需 PM 決策）**、以及多處用語不一致。

最優先回給 PM／設計確認的四件事：

1. 批次通過語意錯誤（原型整案通過，spec 明文禁止）
2. D004 表格切換／關閉防呆整頁缺席
3. 一單多明細 vs 資料契約 A.6 單一 rowId（需決策，牽動閘門粒度）
4. 通過條件「單人核可」顯示邏輯未做

## 三、資料契約對不上（需 PM 決策：改 spec 或改原型）

1. **一單多明細**：審核中心差旅案例「高鐵、住宿、交通共 4 筆明細，總額 NT$ 8,460」「同一單號包成一張單審核」。附錄 A.6 `source = {roomId, tableId, rowId}` 為單一 rowId，裝不下多明細。若採一單多列，A.6 結構與 D002 閘門粒度（逐列狀態）皆須改。另原型自身矛盾：列表寫 4 筆明細，detailRows 僅 3 筆。
2. **工作流／RPA 觸發送審**：原型有申請人＝「系統代理 / Ling 觸發」、來源「工作流 / 正式回信表格」案例。A.6 `applicantId` 為帳號，系統觸發型態 spec 未定義（spec 2 章僅提及流程可套用於 RPA 節點，未定義送審單欄位對應）。
3. **「即將逾期」＋倒數（SLA）**：原型緊迫概念為時限倒數（「剩 6 小時」），spec 為 urgency 兩級 enum（一般／緊急）。兩套語意：原型未做 spec 的緊急度篩選，spec 無逾期概念。
4. **帳號管理整頁**：原型將直屬主管／代理人設定做在「帳號管理」頁（與群組標籤同頁切換），spec 3 章介面表僅列「系統管理 · 群組標籤」。此為填補 spec 缺口——C002 說「提醒具帳號管理權限者補齊」但未定義補在哪。建議 spec 補列此介面或標注沿用既有帳號管理頁。

## 五、其他小項

- 審核中心「我送出的」示範資料缺「審核通過」範例，四態只展示三態（B003）。
- 審核中心篩選為「狀態／來源／搜尋框」，spec B001 要求「申請來源、緊急度、申請人」——緊急度與申請人無獨立篩選。
- 建表精靈空狀態「尚未建立簽核流程」永不觸發（demo flows 固定 4 筆，JS 無切換邏輯），且無導去流程設定的連結；文案「再回到表格設定套用」與 spec 動線「建立後再回來套用」不同。
- 建表精靈權限區 6 檔位（無檢視／個人檢視／個人管理／全部唯讀／團隊協作／全部編輯）：spec 對權限內容零著墨，整塊屬未詳述區，需補規格或由權限模組另行定義。
- 流程設定新增動態角色時 `fallback` 資料值為空字串：下拉靠瀏覽器預設顯示第一項（恰為 spec 預設值），但 hint 文字落到「略過此關」分支，視覺與資料自相矛盾。
- 流程設定動態角色判斷用寫死名稱字串（`dynamicApproverNames` Set）比對，非附錄 A.3 的 type/ref 多型結構——原型層級可接受，RD 實作勿沿用。
- ui-brief 自述有誤：稱流程設定頁「可設定退回上一關或申請人」，實際原型無此配置項（沒有反而正確——spec 中退回是審核中心的審核動作，非流程層級設定）。

## 六、原型有、spec 未涵蓋（合理延伸，建議 spec 補記或標注範圍外)

- 流程設定：複製流程、複製關卡、側欄流程搜尋、從流程端直接「套用到表格」（spec 套用入口在 D001 表格端）、「編輯中」暫態 pill
- 審核中心：狀態篩選器、退回／否決後的「處理結果與原因」檢視模式、前往來源聊天室／表格跳轉
- 群組標籤／帳號管理：批量設定主管、批量調整權限、群組批次刪除、匯出（帳號／群組清單）、「加入部門」一鍵加入、未儲存離開確認、已套用位置前往連結、資源配額／到期日／啟用停用
- 建表精靈：第 4 步「通知設定」（spec 介面表無此頁）、已選流程摘要卡（關卡 chip、已套用 N 張表格）
- 表格設定：基本資訊／欄位管理／權限／通知等其他分頁（框架性質）

## 七、已驗證相符的重點項（抽樣）

- 線性流程：拖曳排序、無平行分支 UI（A001）
- fallback 三選項齊、僅動態角色關卡顯示（A001/C002）
- 單筆否決／退回必填原因，JS 有強制驗證（B001 AC）
- 「我送出的」無撤回鈕、被退回可見原因（B003、範圍邊界）
- 群組已套用禁刪三層防呆（C001、edge 13）
- supervisorStatus 三態 enum 值一致、主管可複數、主次代理人齊且有不可相同驗證（A.5）
- GroupTag status 三值（無成員／未套用／已套用）為衍生計算（A.4）
- 流程狀態 enum 草稿／啟用中／停用（A.1)
- 建表精靈步驟名「權限&審核」、下拉可搜尋流程名稱或關卡、可略過（第 3 章、D001）
- 表格設定下拉搜尋涵蓋名稱＋關卡、無流程引導文案（D001）
