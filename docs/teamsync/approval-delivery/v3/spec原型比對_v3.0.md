# 資料簽核模組 v3.0 — 開發文件 vs UI 原型 比對報告

- 比對日期:2026/07/13
- 比對基準:`資料簽核模組_開發文件_v3.0.md`(2026/07/10)
- 原型:`prototype/` 內 5 個 HTML 原型 + 1 份 UI 說明(ui-brief)
- 方式:逐頁細讀原型(含內嵌 JS 與示範資料),對照開發文件的功能規則與驗收標準(AC)

## 這份報告在講什麼

開發文件是「規則書」,HTML 原型是設計師照規則書做的「樣品」。本報告逐條比對兩者,找出對不上的地方,分四類:

| 分類 | 意思 | 嚴重度 |
| --- | --- | --- |
| 一、行為牴觸 | 原型做出來的行為跟開發文件寫的**相反**。RD 若照原型刻,功能就是錯的 | 高,需改原型 |
| 二、防呆缺席 | 開發文件要求的保護機制,原型完全沒做。RD 看原型會以為不用做 | 高,需補原型或註記 |
| 三、兩邊沒對齊 | 原型做了開發文件沒定義的設計,不是誰錯,但得有人拍板以哪邊為準 | 需 PM 決策 |
| 四、名詞不統一 | 同一個東西兩邊叫法不同。這些字會變成資料庫的狀態值,寫進去難改 | 低,但要開工前處理 |

**結論:大方向一致,可開工;但有四件事要先回 PM 確認**——批次通過語意(1-1)、表格換流程防呆(2-1)、群組成員防呆(2-2)、一單多明細(3-1)。

---

## 一、行為牴觸:原型跟開發文件說的相反

### 1-1. 批次通過的意思做錯了(最嚴重)

**白話**:假設一筆報帳單要過三關(主管→財務→總經理)。開發文件說:主管勾十筆按「批次通過」,意思只是「我這關放行」,這十筆接著要去第二關繼續審。但原型按下去十筆直接變「已通過」——等於主管一個人把三關全跳過。開發文件有明文禁止這種整案通過。

- 原型證據:審核中心 `confirmBatchAction()` 內 `status.textContent = "已通過"`
- 違反:B002 AC「兩筆狀態皆前進至第 2 關(非『已通過』),仍為審核中」、edge case 8

**解答**：以開發文件為主，只限此關放行

### 1-2. 批次否決不用填理由

**白話**:開發文件說否決一定要寫原因(申請人才知道為什麼被打槍),原型的批次否決視窗連填原因的格子都沒有,直接就能送出。

- 原型證據:`openBatchConfirm("reject")` 直接開確認 modal,無原因欄位、未呼叫 `requireReviewReason()`(單筆否決的驗證倒是有做)
- 違反:B002 AC、edge case 9

**解答**：以開發文件為主，批次否決每個都需要原因。

### 1-3. 第一關不該出現「退回上一關」

**白話**:第一關前面沒有關卡,退無可退,開發文件說這選項要藏起來。原型沒藏——示範資料明明在第 1/3 關,退回選單照樣出現。

- 原型證據:審核中心 JS 無任何依關卡序隱藏 `data-return-target="previous"` 的邏輯
- 違反:B001 AC「僅顯示『退回申請人』」、edge case 4

**解答**：第一關退回僅顯示退回申請人

### 1-4. 「單人核可」的切換邏輯沒做

**白話**:開發文件說,一關的審核者如果只有一個人,通過條件就固定是「單人核可」,不給切換;有群組或多人時才出現「任一人 / 所有人」的選項。原型不管幾個人都顯示「任一人 / 所有人」兩顆鈕,而且畫面上根本沒有「單人核可」這個選項——資料裡有這個值,但渲染不出來,使用者還能把單人關卡切成 OR/AND。

- 原型證據:流程設定 `renderNode()` 對每關固定渲染兩鈕 segmented
- 違反:A001 AC、edge case 17、附錄 A.2 condition enum(UI 只呈現 2 值,文件定義 3 值)

**解答**：依開發文件

### 1-5. 動態角色「部門主管」選不到

**白話**:開發文件定義四種動態角色(直屬主管、職務代理人、次要職務代理人、部門主管),原型的選擇清單只有前三種。「部門主管」只出現在預先塞好的示範資料裡,使用者自己選不到。

- 原型證據:流程設定 `approverCandidates` 候選清單缺「申請人的部門主管」
- 違反:A001

**解答**：沒有部門主管設定，依UI

## 二、防呆缺席:文件要求的保護機制,原型零實作

### 2-1. 表格換流程 / 關閉審核,沒有任何保護(D004)

**白話**:如果一張表格還有單子正在審,這時換掉流程或關掉審核,審到一半的單子會變孤兒。開發文件要求:有「審核中」資料時要鎖住、不准換、提示先結清。原型那頁隨便你換——點任何流程都直接切換成功,連「啟用審核」的開關都是畫假的(純 CSS,點了沒反應)。這是「表格設定 · 審核」這一頁存在的核心理由,卻完全沒做。

- 原型證據:`selectFlow(id)` 無條件切換;flows 資料無審核中筆數欄位;全檔搜不到「審核中」「結清」字樣
- 違反:D004 兩條 AC、edge case 12

**解答**：確保所有資料都已經完結，不可有審核中的資料才可更改流程。

### 2-2. 移除群組成員,不檢查會不會害單子卡死(edge 14)

**白話**:假設「財務審核」群組只剩一個人能審某張單,把他移出群組,那張單就永遠沒人審。開發文件要求移人 / 刪群組前要檢查並阻擋,原型直接讓你移,連對應的資料欄位(`affectsRunning`)都沒有。註:另一個防呆「已套用的群組不可刪除」倒是做得很完整,三層都擋。

- 原型證據:`data-remove-member` handler 直接 `filter` 掉成員,無任何檢查
- 違反:C001 D2、edge case 14、附錄 A.4

**解答**：依開發文件為主

### 2-3. 兩人同時審一筆的處理,沒有呈現(edge 2)

**白話**:一關設「任一人核可」時,A 先核准、B 晚一步想否決,開發文件說 B 要被擋下並提示「已由 A 處理」。原型完全沒有這個情境——全檔搜不到「已由」兩字。

- 違反:5.3、edge case 2

**解答**：依開發文件為主

### 2-4. 流程的「使用中不可刪改」只做半套

**白話**:開發文件說流程只要「已套用到表格」或「有單子在跑」就要鎖住,不准刪改。原型有畫出鎖定的樣子(鎖頭標籤、欄位反灰),但:(a) 三條示範流程全都沒鎖,其中兩條明明已套用到表格、照文件應該要鎖;(b) 鎖定的說明文字只提「有待審資料」,漏了「已套用」這個條件;(c) 側欄的「刪除流程」完全不檢查鎖定,照樣能點。

- 違反:A002、附錄 A.1(appliedTo 非空 或 isRunning ➔ locked)

**解答**：依開發文件為主

## 三、兩邊沒對齊:需 PM 拍板(改文件或改原型)

### 3-1. 一張單多筆明細(最需要決策)

**白話**:原型做了「差旅費一張單裡包高鐵+住宿+計程車多筆明細,一起審」的設計,很貼近實際報帳情境。但開發文件的資料結構一張送審單只對應**一筆**資料(`source = {roomId, tableId, rowId}`,單一 rowId),裝不下多筆。要支援就得改資料契約,連帶影響資料狀態閘門是逐筆還是逐單。另外原型自己也矛盾:列表寫「共 4 筆明細」,明細區只有 3 筆。

- 影響:附錄 A.6、D002 閘門粒度

**解答**：再討論是否可執行

### 3-2. 系統(工作流)自動送審

**白話**:原型有一筆申請人是「系統代理 / Ling 觸發」的單,來源是工作流。開發文件的申請人欄位定義是人的帳號,系統觸發這種型態沒定義(文件只提過流程可套用於 RPA 節點,沒定義送審單欄位怎麼填)。

- 影響:附錄 A.6 applicantId、source

**解答**：RPA再規劃

### 3-3. 「即將逾期」倒數 vs「一般/緊急」

**白話**:原型的急迫概念是時限倒數(「剩 6 小時」),開發文件是兩級標籤(一般/緊急)。兩套不同的東西:原型沒做文件要的緊急度篩選,文件沒有逾期概念。要哪一套、還是都要,需決定。

- 影響:B001 篩選、附錄 A.6 urgency

**解答**：先不用做

### 3-4. 「帳號管理」整頁是文件沒列的介面

**白話**:原型把直屬主管、職務代理人的設定做在「帳號管理」頁(與群組標籤同頁切換),開發文件的介面清單只列了「群組標籤」。但這其實是補了文件的洞——文件說主管沒設定時要「提醒具帳號管理權限者補齊」,卻沒說去哪補。建議文件補列這個介面,或註明沿用既有帳號管理頁。

- 影響:第 3 章介面表、C002

**解答**：以現有+UI介面為主

## 四、名詞不統一(開工前統一,這些字會進資料庫)

| 原型用語 | 開發文件用語 | 出處 |
| --- | --- | --- |
| 退回上一**層** | 退回上一**關** | 審核中心退回選單 |
| **簽核**中 | **審核**中 | 審核中心「我送出的」狀態值 |
| **簽核**流程設定、啟用**簽核** | **審核**流程設定 | 建表精靈文案(該頁自身也簽核/審核混用) |
| 尚未填寫 / 無直屬主管 | 尚未設定 / 確認無主管 | supervisorStatus 中文顯示(英文代碼 assigned/none/unset 兩邊一致) |
| 送出時間 | 申請時間 | 審核中心詳情 |
| 通過 | 核准通過 | 審核中心動作鈕 |
| 啟用 | 啟用中 | 流程設定側欄狀態標籤(下拉選項本身正確) |
| 不可送出,提醒補齊對應人 | 不可送出(提醒補齊對應人) | fallback 選項,措辭微差 |

## 五、其他小項

- 審核中心「我送出的」示範資料缺「審核通過」的例子,四種狀態只展示了三種(B003)。
- 審核中心的篩選是「狀態 / 來源 / 搜尋框」,文件要求「申請來源、緊急度、申請人」——緊急度與申請人沒有獨立篩選(B001)。
- 建表精靈的「尚未建立簽核流程」空狀態永遠不會出現(示範資料固定 4 筆流程,JS 沒有切換邏輯),而且只有文字、沒有連去流程設定的連結;文案寫「再回到表格設定套用」,文件的動線是「建立後再回來(精靈)套用」。
- 建表精靈的權限區有 6 個檔位(無檢視~全部編輯),開發文件對權限內容完全沒著墨,整塊屬未定義區,需補規格或由權限模組另行定義。
- 流程設定新增動態角色時,fallback 的資料值是空字串:下拉靠瀏覽器預設顯示第一項(剛好是文件的預設值),但提示文字卻落到「略過此關」的分支——畫面跟資料自相矛盾。
- 流程設定判斷「是不是動態角色」是用寫死的名稱字串比對,不是文件附錄 A.3 的 type/ref 結構——原型示意可接受,RD 實作勿沿用(改個顯示名稱邏輯就壞)。
- ui-brief 自己的描述有一處錯:說流程設定頁「可設定退回上一關或申請人」,實際原型沒有這個設定項。沒有反而是對的——文件裡「退回」是審核中心的審核動作,不是流程層級的設定。

## 六、原型有、文件沒提的功能(合理延伸,建議文件補記或標注範圍外)

- 流程設定:複製流程、複製關卡、側欄流程搜尋、從流程端直接「套用到表格」(文件的套用入口在表格端 D001)、「編輯中」暫態標籤
- 審核中心:狀態篩選器、退回/否決後的「處理結果與原因」檢視、前往來源聊天室/表格的跳轉
- 群組標籤/帳號管理:批量設定主管、批量調整權限、群組批次刪除、匯出、「加入部門」一鍵加入、未儲存離開確認、已套用位置的前往連結、資源配額/到期日/啟用停用
- 建表精靈:第 4 步「通知設定」(文件介面表無此頁)、已選流程摘要卡
- 表格設定:基本資訊/欄位管理/權限/通知等其他分頁(框架性質)

## 七、驗證過沒問題的重點項

- 線性流程:拖曳排序、無平行分支 UI(A001)
- fallback 三選項齊全、只在動態角色關卡顯示(A001/C002)
- 單筆否決/退回必填原因,JS 有強制驗證(B001 AC)
- 「我送出的」沒有撤回鈕、被退回可看原因(B003、範圍邊界)
- 群組「已套用不可刪除」三層防呆(C001、edge 13)
- supervisorStatus 三態代碼一致、主管可設多位、主/次代理人齊全且有不可相同驗證(附錄 A.5)
- 群組狀態三值(無成員/未套用/已套用)為自動計算(附錄 A.4)
- 流程狀態:草稿/啟用中/停用(附錄 A.1)
- 建表精靈:步驟名「權限&審核」、下拉可搜尋流程名稱或關卡、可略過(第 3 章、D001)
- 表格設定:下拉搜尋涵蓋名稱+關卡、無流程時的引導文案(D001)
