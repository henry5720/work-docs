# 10 FAQ 技術提問回覆

本文件彙整 2026/03/31 客戶 IT 部門之技術反饋，並提供正式官方回覆。

### Q1: 缺乏資料修改與刪除機制？
**回覆**：系統目前已支援單筆量測資料之 `PUT` (修改) 與 `DELETE` (刪除) 接口。批量接口目前定位為數據採集 (Append Only)，批量修正功能已納入 v1.2 產品路徑圖。

### Q2: 數據重複性處理邏輯不明確？
**回覆**：系統現階段採用 **Append (追加)** 模式。判定唯一性之業務鍵組合為「計畫名稱 + 管制項目 + 批號 + 樣本索引 (idx)」。重複上傳同一批號會被視為新的樣本批次。

### Q3: 計畫管理與變更的彈性不足 (n=3 變 n=5)？
**回覆**：為確保 SPC 管制圖之統計公式一致性，單一計畫之樣本數 (n) 於建立後即固定。若製程規範異動 (n=3 → n=5)，系統建議建立新版本計畫，以正確應用 $d_2, c_4$ 等統計常數。

### Q4: 欄位規範與系統限制細節不全？
**回覆**：
- **長度限制**：核心字串欄位 (`part_number`, `batch_number`, `characteristic_name`) 最大長度均為 **128 字元**。
- **特殊符號**：`characteristic_name` 禁入 `/ \ : * ? " < > |` 等檔案路徑敏感字元。
- **頻率限制**：詳見「01 需求規格書」，每分鐘上限 60 次請求。

### Q5: 任務結果追蹤之關聯性 (ID 對應)？
**回覆**：API 規格已新增 `ref_id` 欄位。客戶可於匯入時帶入原始編號，系統處理完畢後將於 Task Status 返回 `ref_id` -> `sample_id` 之對照表。

### Q6: 錯誤處理、重試機制與 TTL？
**回覆**：
- **任務狀態保留 (TTL)**：已從 1 小時延長至 **24 小時**。
- **重試建議**：對於 502/504 類錯誤，建議採取 **指數退避 (Exponential Backoff)** 之重試策略。
