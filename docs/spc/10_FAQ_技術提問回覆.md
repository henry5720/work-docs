# 10 FAQ 技術提問回覆

本文件彙整 SPC 系統對接之常見技術提問、業務邏輯與系統限制，內容包含針對客戶最新回饋之說明。

---

## 1. 數據異動與維護

### Q1: 系統是否支援資料的修改與刪除？
**事實**：支援。雖然 `all-in-one` 接口以「新增」為主，但系統提供標準的維護接口：
- **修改**：可透過 `PUT` 接口更新特定 `sample_id` 的數值或 `ccm_id` 的配置資訊。
- **刪除**：提供 `DELETE` 接口進行 **硬刪除 (Hard Delete)**。
- **建議**：若發現批次匯入錯誤，建議先透過 `DELETE` 刪除該計畫或樣本，修正後重新匯入。詳細端點請參閱 [08 API 規格與用法說明](./08_API_規格與用法說明.md)。

---

## 2. 數據重複性與唯一性

### Q2: 若重複上傳相同批號 (batch_number) 的資料，系統如何處理？
**事實**：系統採 **追加 (Append)** 模式。
- 若同一個 `batch_number` 的資料被重複發送兩次，系統會產生兩筆重複的樣本記錄。
- 系統**不會**自動進行內容級別的去重 (De-duplication)，客戶端應自行控管發送邏輯。

### Q3: 系統中的「唯一鍵 (Unique Key)」是如何定義的？
**事實**：
- **計畫識別**：由 `tenant_id` 與 `CCM Name` (層別組合名稱) 決定唯一性。
- **項目識別**：在同一計畫下，以 `characteristic_name` 作為唯一標識。
- **樣本識別**：在同一項目下，以自動遞增的 `idx` (樣本索引) 區分。
- **結論**：相同的 `管制項目 + 料號 + 批號`，若存在於不同的線別或廠區（且線別/廠區有參與命名），系統會將其視為隸屬於不同計畫的獨立數據。

## 3. 計畫管理與變更

### Q4: 若抽樣標準改變（例如從 n=3 改為 n=5），該如何處理？
**事實**：為了確保統計管制圖（如 X̄-R 圖）的常數 $d_2, c_4$ 計算一致，系統要求同一計畫下的樣本數量必須完全一致。
- **限制**：若現有計畫為 n=3，直接匯入 n=5 的資料會觸發 `inconsistent sample sizes` 錯誤。
- **解決方案**：若需變更抽樣數，必須先透過 **計畫管理 API (PUT)** 或 **Web 介面** 調整該計畫的抽樣設定，或建立一個新的計畫名稱（調整命名層別組合）以適應新規格。

---

## 4. 系統限制與規格

### Q5: 欄位長度與格式有什麼具體限制？
**事實**：
- **字串長度**：`part_number`、`batch_number`、`characteristic_name` 與層別鍵值對均限制為 **1, 128** 字元。
- **特殊符號**：支援大多數標準字元，但建議避開路徑敏感字元（如 `/ \ : * ? " < > |`）以確保報表匯出相容性。

### Q6: API 是否有頻率限制 (Rate Limit)？
**事實**：
- **單次請求**：技術上限為 **10,000 筆** items。
- **頻率限制**：後端目前未設置硬性的 Rate Limit，但建議每批次間隔 1 秒以上，並採用指數退避 (Exponential Backoff) 處理可能的 429 或 503 錯誤。

## 5. 任務追蹤與錯誤處理

### Q7: 如何將回傳的 ID 與原始提交的資料進行一一對應？
**事實**：目前採 **「扁平化統計 (Flat Statistics)」**。
- 任務結果僅回傳該批次成功產出的 ID 列表，不保證與原始 `items` 陣列的索引對應。
- **建議**：若開發者需精確追蹤每一筆資料的 ID，建議將請求拆分為更小的批次。

### Q8: 任務狀態 (Task Status) 的有效期多久？
**事實**：固定為 **3,600 秒 (1 小時)**。
- 這是為了避免 Redis 內存過度膨脹。超時後查詢將回傳 `404 Not Found`。系統不提供延長保存或歸檔查詢功能，請於匯入完成後立即記錄結果。

### Q9: 為什麼樣本值一定要用字串格式？
**事實**：為了確保精確度並 **自動推斷精密度**。
- 系統會從字串內容推斷量測儀器的原始位數（如 `"1.250"` 代表 3 位小數）。
- 使用數值型態 (Float) 可能因 JSON 解析產生微小誤差（如 `1.24999999`），進而影響 Cp/Cpk 統計結果。

### Q10: 如果部分資料匯入失敗，會發生什麼？
**事實**：系統採 **「計畫級別事務 (CCM-Level Transaction)」**。
- 匯入以計畫為分組。若某一計畫內有資料不符規範，**僅該計畫**的所有異動會回滾。
- 其他符合規範的計畫仍會成功寫入，並記錄在 `created_ccm_ids` 中。
- 前端目前實作有 3 次重試機制（指數退避）。
